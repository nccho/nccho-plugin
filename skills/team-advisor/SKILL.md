# Team Advisor

## 0. 전제 조건 및 규모 확인

1. `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` 환경변수가 켜져 있는지 확인한다. 꺼져 있으면 settings.json의 `env`에 추가하는 방법을 안내하고 여기서 중단한다 (재시작 필요).
2. 현재 세션이 이미 다른 팀에 속해 있으면(nested team 불가) 사용할 수 없다고 안내하고 중단한다.
3. **요청이 짧거나 맥락이 애매하면(예: "비교 자문 해줘"처럼 대상/기준이 불분명한 경우) 바로 팀을 스폰하지 않는다.** agent teams는 팀원마다 독립적인 context window를 쓰기 때문에 토큰 비용이 크다. 이런 경우 아래를 먼저 확인한다:
   - 자문 대상(무엇에 대한 결정/비교인지)이 명확한지
   - 이 정도 비용을 들일 만큼 중요한 의사결정인지
   - 불명확하면 AskUserQuestion 등으로 짧게 되물은 뒤 진행한다. 단순 질문으로 판단되면 팀을 구성하지 말고 main agent가 직접 답한다.

## 1. 관점 설계

요청에 맞는 2~4개의 상호 배타적인 독립적 관점을 정의한다. 예:
- 기술적 실현 가능성 (feasibility)
- 리스크 / 보안
- 사용자 영향 / UX
- 비용 / 유지보수

## 2. 팀 구성 (자연어 트리거)

전용 도구 없이 자연어 지시로 팀을 스폰한다. 예:

> "Spawn N teammates to independently advise on <주제>: one from <관점1> angle, one from <관점2> angle, ... Each must reach their own conclusion with reasoning, without seeing each other's answers."

팀원 간 결론을 미리 공유하지 않도록 명시해 독립성(편향 방지)을 유지한다. 팀원 수는 3명을 기본으로 하고, 관점이 많아도 5명을 넘기지 않는다(coordination overhead 고려).

## 3. Main agent 자체 자문

팀원들과 별개로 main agent 자신의 추천과 근거도 정리해 비교 대상에 포함시킨다.

## 4. 종합 비교

팀원 전원의 결론 + main agent의 결론을 표(관점 / 추천 / 핵심 근거 / 리스크)로 정리한다.
- 의견이 갈리는 지점을 명시적으로 강조한다.
- 다수결이 아니라 근거 기반으로 최종 추천 1개를 제시한다. 소수 의견이 더 설득력 있으면 그것을 채택하고 이유를 설명한다.

## 5. 마무리

팀을 정리(팀 리더가 종료)하고, 사용자에게 의견이 갈린 지점과 최종 추천·근거를 간결하게 보고한다.
