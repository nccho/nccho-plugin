# Dev Flow

코드 수정을 "구현하고 끝"이 아니라 **브랜치 → 테스트 → 구현 → 정리 → 리뷰 →
MR → 승인**의 파이프라인으로 진행한다. 각 단계는 통과 조건이 있으며, 통과하지
못하면 다음 단계로 넘어가지 않는다. 마지막 승인은 항상 사용자 몫이다.

## 0. 완료 기준 확인

작업의 완료 기준이 참/거짓으로 판정 가능한 수준으로 명확한지 먼저 본다.

- **흐릿하면**: `goal-interview` 스킬을 먼저 실행해 완료 기준 체크리스트를
  확정한다. 인터뷰가 끝나며 제안되는 `/goal <조건>` 한 줄을 사용자가 등록하면
  이후 파이프라인 전체가 그 조건 아래에서 돈다 (조기 종료 방지).
- **이미 명확하면**: 인터뷰 없이 완료 기준을 한 문장으로 복명복창만 하고
  바로 1단계로 간다.

## 1. 브랜치 생성

- **main/master에서 직접 수정하지 않는다.** 작업 전 반드시 브랜치를 만든다.
- 브랜치 이름은 저장소의 기존 관례를 따른다 (`git log`로 확인 — 예: `feat/`,
  `fix/`, `docs/` 접두사). 관례가 없으면 `feat|fix/<간결한-작업명>`.
- 통과 조건: 새 브랜치로 체크아웃된 상태 (`git branch --show-current`).

## 2. 테스트 코드 작성 (필요시)

- 작성 기준: **동작 변경이 있고, 프로젝트에 테스트 인프라가 존재하면** 작성한다.
  가능하면 구현 전에 실패하는 테스트를 먼저 만든다 (기준의 실행 가능한 형태).
- 건너뛰는 경우: 테스트 인프라가 없는 프로젝트, 동작이 바뀌지 않는 변경.
  건너뛸 때는 그 사실과 이유를 한 줄로 알린다 — 조용히 생략하지 않는다.
- 통과 조건: 새 테스트가 실행되고, 구현 전이므로 실패하는 것을 확인.

## 3. 코드 수정

- **위임 기준 (기본: 직접 구현)** — 메인 에이전트가 직접 구현하는 것이 기본이다.
  단, 완료 기준이 확정되어 있고 판단 여지가 없는 기계적 구현(대량 반복 수정,
  정해진 패턴 적용)은 `implementer` 에이전트에 위임해도 된다. 구현 중 사용자
  확인이 필요할 수 있는 작업은 위임하지 않는다 — 서브에이전트는 실행 중 왕복이
  안 된다.
- 설계가 갈리는 지점을 만나면 임의로 정하지 말고, 사소하면 선택 근거를 남기고
  진행, 결과에 영향이 크면 사용자에게 묻는다.
- 통과 조건: 2단계의 테스트가 통과하고, 기존 테스트도 깨지지 않음.

## 4. 코드 정리 — /simplify

- 번들 `/simplify` 스킬을 호출해 이번 변경분의 중복·과잉 복잡도를 정리한다.
- 정리 후 테스트를 다시 실행해 동작이 유지되는지 확인한다.
- 통과 조건: simplify 반영 후에도 전체 테스트 통과.

## 5. 리뷰 — /code-review

- 번들 `/code-review` 스킬을 호출해 이번 diff의 버그·품질 이슈를 리뷰받는다.
- findings가 나오면 반영하고, 반영분이 크면 테스트 재실행으로 확인한다.
  반영하지 않기로 한 finding은 이유와 함께 사용자에게 보고한다.
- 통과 조건: findings가 모두 반영되었거나 명시적으로 보류 처리됨.

## 6. MR/PR 생성 — 승인 게이트

- **생성 전에 멈춘다.** MR/PR 생성은 외부에 드러나는 행위이므로, 변경 요약과
  MR 제목·본문 초안을 보여주고 사용자 승인을 받은 뒤에만 생성한다.
- 커밋 메시지·MR 본문은 저장소 관례를 따른다 (`git log`, 기존 PR 참고).
- 통과 조건: 사용자가 승인했고 MR/PR이 생성되어 링크를 보고함.

## 7. MR 승인 — 여기서 정지

- **MR 승인·머지는 사용자의 몫이다.** 스킬은 MR 링크 보고로 역할이 끝난다.
  승인을 대신 누르거나 self-merge하지 않는다.
- 보고 시 0단계의 완료 기준 체크리스트를 다시 보여주고, 각 기준의 충족 여부를
  판정 결과(검증 명령 + 실제 출력)와 함께 정리한다. "완료 후 직접 확인" 목록이
  있으면 여기서 다시 상기시킨다.

## 예외 처리

- 파이프라인 중간에 사용자가 방향을 바꾸면(스코프 변경), 0단계로 돌아가 완료
  기준부터 갱신한다 — 기준이 낡은 채로 계속 달리지 않는다.
- 단계 실패가 반복되면(테스트가 계속 깨지는 등) 같은 시도를 되풀이하지 말고,
  현재 상태·시도한 것·막힌 지점을 정리해 사용자에게 보고한다.
