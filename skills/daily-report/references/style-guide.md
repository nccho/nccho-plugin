# 일일 업무보고서 스타일 가이드

웹 리서치(한국 회사 보고서 작성법 + 개발/애자일 진행보고)와 사용자 상위권자의 요구사항을 종합한 작성 기준. 모든 일일 보고서는 이 가이드를 따른다.

## 0. 상위권자가 원하는 핵심 5요소 (최우선)

보고 내용에는 아래가 (해당되면) 반드시 담겨야 한다.

1. **왜 이 일을 했는가** — 무슨 문제/배경이 있었는지
2. **무엇을 어떻게 했고, 효과는 무엇인가** — 조치와 결과
3. **(길고 복잡한 일이면) 전체 진행 중 현재 단계** — 무엇을 했고 무엇이 남았는지
4. **일정** — 정해진 마감이 있다면 언제까지인지
5. **장애물** — 진행을 막는 것이 있다면 무엇인지

## 1. 기본 골격: 두괄식 (결론 먼저)

- 핵심 결론을 문단·항목의 **맨 앞**에 두고 근거·과정을 뒤에 붙인다.
- 상사는 빠른 판단이 필요하므로 결론이 먼저 보여야 한다 (보고시간 단축·피드백 명확화 효과 확인됨).
- 실행 팁: (1) 맨 위 한 문장 요약, (2) 각 항목 첫 문장을 주장으로 시작, (3) 제목 자체를 결론으로.
- 출처: https://www.wbcb.co.kr/news/articleView.html?idxno=91326

## 2. 성과 중심 서술 (활동 나열 금지)

- "무엇을 했다"만 적으면 결과가 안 보여 상사가 되묻게 된다. **결과·효과를 수치로** 붙인다.
- 성과가 없어도 "무엇을 배웠는지/다음에 어떻게 할지"를 남긴다.
- Before: "동아리 연합 공연을 기획했다"
- After: "첫 연합공연을 기획하여 기존 공연 대비 관객 수 300% 증가"
- 개별 성과는 **STAR**(Situation 상황 → Task 과제 → Action 행동 → Result 결과)로 구조화하되, 하루 보고에서는 비중 있는 성과 1~2건에만 4단계를 다 쓴다.
- 출처: https://www.haijob.co.kr/blog/writing-method-using-star-methodology/

## 3. 형식: 서술형 개조식 (개조식 + 서술식 혼합)

사용자가 원하는 혼합 형태. 기본은 개조식, 필요한 곳만 서술.

- **개조식이 적합** (`~함/~임/~했음` 명사형 종결, 글머리표): 완료 업무, 진행 현황, 수치 결과, 항목 나열·비교.
- **서술식이 적합**: 배경·맥락 설명, 상황 묘사 등 "글로 풀어야 이해되는" 부분.
- 권장 패턴: 머리에 짧은 상황 설명(서술 한 줄) → 핵심은 항목화(개조식).
- 비효율(장황한 서술): "고객사 요청으로 자료를 다시 정리하였고 내부 검토를 진행하였으며 현재 수정 중입니다."
- 효율(서술형 개조식): 결론=자료 수정 진행 중 / 진행=초안 80% 완료 / 다음=금일 18시 전달 예정
- 출처: https://brunch.co.kr/@itrendlab/29 , https://www.fanruan.com/ko-kr/blog/business-report

### 3-1. 구구절절 방지 (서술이 길어지지 않게)

서술을 허용하되 **한 항목이 만연체로 부풀지 않도록** 아래를 지킨다.

- **한 항목 = 한 요점.** 배경·조치·효과를 한 문장에 다 담지 말 것. 배경이 꼭 필요하면 앞에 짧은 한 줄로 떼고, 결과는 개조식으로 나눈다.
- **종속절 이어붙이기 금지.** `~했으며`, `~했고`, `그 과정에서 ~해` 로 문장을 계속 잇지 않는다. 한 항목은 **2문장 이내**, 접속으로 3절 이상 잇지 않는다.
- **곁가지는 버린다.** "여러 관점 자문을 두 차례 거쳐", "~할 위험이 있었음" 같은 과정 수식·가정은 결과에 영향 없으면 생략.
- 화살표(`→`)나 `/`로 **배경 → 조치 → 효과**를 끊어 쓰면 종속절보다 잘 읽힌다.
- Before(구구절절): "전날 정정한 검증 규칙이 명세 작성·검증 단계에 아직 반영돼 있지 않아 CDM이 없는 정상 항목이 근거 없이 반려될 위험이 있었음. 두 단계를 CDM 존재 여부로 분기하도록 확장해 상·하류 규칙을 일치시켰고, 여러 관점 자문을 두 차례 거쳐 설계를 확정함."
- After(간결): "하류(작성·검증) 단계에 전날 규칙이 미반영 → CDM 유무로 분기해 상·하류 규칙 일치. 정상 항목 오반려 위험 해소."
- 결론 한 줄 요약(§1)도 접속으로 잇지 말 것. 성과 1~2개만 골라 진짜 한 줄로.

## 4. 진행 상황 표현 (긴/복잡한 작업)

3층 구조로 "전체 중 어디쯤인지"를 보여준다.

- **마일스톤**: 큰 흐름의 체크포인트 (실질 성과만; 남발 금지)
- **% 진행률**: 완료 근접도
- **체크리스트**: 세부 완료/남은 항목
- 표기 예: `전체 4단계 중 2단계 진행 (약 50%) / 완료: 설계·API / 남은 일: 연동·테스트`
- 출처: https://www.teamgantt.com/blog/the-how-and-why-of-using-milestones-in-your-project-plan , https://asana.com/resources/milestone-chart

## 5. 장애물 보고: CLEAR 프레임워크

### 5-1. 먼저 판단: 이 섹션에 쓸 일인가

`■ 장애물·요청`은 **지금 진행을 막고 있으면서, 동시에 상사/동료의 조치나 지원이 필요한** 것만 넣는다.

- 판단 기준 한 줄: "상사가 이걸 읽고 **할 일이 생기는가?**" → 아니오면 장애물이 아니다.
- 이미 스스로 해결했거나 / 내가 처리 중이라 지원 요청이 없거나 / 단순 후속 확인만 남은 일은 이 섹션에 쓰지 않는다. → `■ 오늘 완료`의 해당 항목 끝에 **한 줄 후속 메모**로 붙인다.
- 지원 요청(아래 A)이 비면 그건 장애물이 아니라는 신호다. "별도 지원 요청 없음" 같은 문구로 슬롯을 채우지 말 것.

### 5-2. 쓸 때: CLEAR로 빠짐없이, 단 태그 없이 자연 문장으로

CLEAR는 요소를 빠뜨리지 않기 위한 **작성자용 체크리스트**다. `[C]`·`[L]` 같은 태그를 출력에 남기지 말고, 아래 요소를 자연스러운 한두 문장에 녹인다. 해당 없는 요소는 생략한다.

- **C**larify: 무엇이 막는지
- **L**ist attempted: 이미 시도한 것
- **E**xplain impact: 일정/프로젝트에 미치는 영향
- **A**sk: 필요한 구체적 지원 (이게 없으면 5-1로 돌아가 이 섹션에서 뺀다)
- **R**eport back: 이후 결과 공유 약속
- 예시(태그 없이): "벤더 API 문서 대기로 연동이 막혀 있음. 두 차례 팔로업·포털 확인했으나 문서 없이는 출시 지연 우려. 이번 주 벤더 에스컬레이션 지원 필요, 회신 오면 즉시 공유 예정."
- 같은 blocker가 3~4일 연속 움직임 없으면 보고서에 반복 기재하지 말고 직접 에스컬레이션.
- 출처: https://codesignal.com/learn/courses/driving-projects-through-leadership-and-influence/lessons/eliminating-blockers-and-risks , https://www.atlassian.com/agile/scrum/standups

## 6. 일정/데드라인 커뮤니케이션

- **조기 통보**(가능성 보이면 즉시, 막판 X) → **해결 중심**("왜 늦었나"보다 "어떻게 정상화") → **무비난**.
- 전달 내용: 무엇이 언제까지인지 + 지연 시 영향 + 수정된 계획(새 일정/완화책).
- 출처: https://hubstaff.com/blog/communicating-project-delays/

## 7. Do / Don't 요약

**Do**: 결론 먼저 · 성과·수치 명시 · 완료/진행/계획 구분 · 장애물엔 해결방향 동반 · 한눈에 이해되고 추가 질문 필요 없게.

**Don't**: 결론 없이 과정만 장황 · "잘 진행 중" 같은 모호한 표현 · 이슈만 적고 대책 없음 · 다음 계획 누락 · 불필요한 세부 과다 · 지원 필요 없는 일을 장애물·요청에 넣기 · CLEAR를 `[C][L]…` 태그로 출력 · 한 항목에 종속절을 3절 이상 이어붙인 만연체.

## 8. 출력 템플릿 (기본형)

```
[결론 한 줄 요약]

■ 오늘 완료
- (배경 서술 한 줄) …문제가 있어, ~를 ~하여 ~효과를 냄
- …함 / …임

■ 진행 중  (전체 N단계 중 M단계 · 약 %)
- 완료: …  |  남은 일: …

■ 장애물·요청  (지원이 필요한 진짜 blocker만 — §5-1 판단 먼저)
- (막힌 것 · 시도한 것 · 영향 · 필요한 지원 · 후속 계획을 태그 없이 한두 문장으로)

■ 일정  (해당 시)
- ~작업 M/D 마감 예정 (리스크: …)
```

간단한 일상 업무는 `■ 오늘 완료`만으로도 충분하다. 항목은 해당될 때만 쓴다.
