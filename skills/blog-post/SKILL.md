# Blog Post Writer (nclovehs.blogspot.com)

사용자의 개인 Blogger 블로그(https://nclovehs.blogspot.com)에 올릴 글을 기존 문체와 최대한 비슷하게 작성한다. 카테고리별 문체 차이가 크므로 반드시 `references/style-guide.md`의 해당 카테고리 프로필을 따른다.

## 0. 카테고리 판별

사용자 요청에서 카테고리를 판별한다. 모호하면 물어본다.

- **프로그래밍/기술**: SSH, Git, 에러 해결, 툴 세팅 등 개발 관련 주제
- **여행**: 일본 여행(교토/오사카 등)이 실제로 사용자가 쓴 여행 글의 기준. 다른 지역 여행 글을 요청받으면 톤은 동일하게 적용하되, 일본 여행 글을 참고 기준으로 삼는다는 점을 사용자에게 짚어준다.
- **맛집**: 특정 식당/카페 리뷰

## 1. (선택) 최신 문체 확인

가능하면 해당 카테고리의 최근 글 1개를 `WebFetch`로 다시 확인해 문체가 바뀌지 않았는지 가볍게 검증한다. 실패하거나 시간이 없으면 `references/style-guide.md`에 정리된 내용만으로 진행해도 된다. 이미 최근에 같은 카테고리 글을 확인했다면 다시 fetch할 필요 없다.

## 2. 초안 작성

`references/style-guide.md`의 카테고리별 구조/어투/이모지/정보 표기 규칙을 그대로 따라 본문을 작성한다.

- 프로그래밍 글: 문제 제기 → 원인(🤔) → 단계별 해결(번호+이모지) → 코드블럭(줄마다 주석) → 마무리 인사/다음 글 예고
- 여행/맛집 글: 서두·마무리 인사 없이 바로 총평/요약으로 시작 → 상세 정보 → 태그로 종료. 존댓말과 반말(팁·감탄 표현)을 자연스럽게 섞는다.

## 3. 개인정보 노출 금지 (필수 확인)

**절대 규칙**: 사용자의 실제 이메일, GitHub/서비스 아이디, SSH 키 지문(fingerprint), API 키/토큰, 전화번호, 실제 집 주소, 차량 번호판 등 개인 식별 정보를 글 본문에 그대로 쓰지 않는다.

- 터미널 로그/스크린샷 등 실제 세션 내용을 근거로 기술 글을 쓸 때는 반드시 더미 값으로 치환한다.
  - 이메일 → `example@email.com`
  - 사용자명/계정명 → `octocat`, `사용자명` 등
  - 키 지문/토큰/해시 → `SHA256:XXXX...` 같은 자리표시자
- 여행/맛집 글에서 특정 개인(동행자 실명 등)을 특정할 수 있는 정보가 섞여 있다면 익명화하거나 생략한다.
- 초안을 내놓기 전에 이 체크를 스스로 한 번 수행하고, 치환한 항목이 있다면 사용자에게 짧게 알려준다.

## 3.5 하단 AI 고지 문구 (필수)

모든 글은 본문 **맨 마지막(태그보다 아래)**에 AI 작성 고지 한 줄을 넣는다. 근거 자료 성격에 맞춰 문구를 고른다.

- **맛집/여행**(실제 방문 기반): `이 글은 실제 방문 내용을 기반으로 AI를 활용해 작성했습니다.`
- **프로그래밍/기술**(방문 개념이 없음): `이 글은 실제 경험을 기반으로 AI를 활용해 작성했습니다.`

한 줄 위에 구분선(`---`)이나 옅은 표기로 본문과 분리하면 자연스럽다. 문구는 임의로 바꾸지 말고 위 형식을 유지한다.

## 4. 출력 형식

다음 네 가지를 항상 함께 제공한다.

1. **제목** (카테고리 스타일에 맞는 대괄호/이모지 포함 형식)
2. **본문 초안** (마크다운, Blogger에 바로 붙여넣을 수 있는 형태 — 3.5의 하단 고지 문구 포함)
3. **퍼머링크(URL slug)**: 영문 kebab-case, 핵심 키워드/지명/음식명/에러 메시지 조합. 기존 슬러그 패턴 예시는 style-guide.md 참고
4. **검색 설명(메타 디스크립션)**: 150자 이내, 핵심 키워드 포함 실용 요약문

## 5. 마무리

톤/분량 조절, 사진 삽입 위치 표시, 추가 정보 보강 등 다음 조정 여지가 있으면 짧게 물어본다.
